一、JupyterHub 简介

​ JupyterHub总的来说为用户群体提供了notebook功能，具体说来它允许用户访问计算环境和资源，而不会给用户带来安装和维护任务的负担。用户可以在共享资源中自己的workspace里完成自己的工作，这些共享资源可以被系统管理员高效管理。

​ JupyterHub 可以全跑在物理机上，也可以上云，它使世界上任何用户都能够为预先配置的数据科学环境提供服务。它是可定制和可扩展的，适合各种团队、学术课程和大型基础设施。

​ 一些优势：

  • Customizable。JupyterHub可用于服务各种环境。它支持Jupyter服务器的数十个内核，并可用于提供多种用户界面，包括JupyterNotebook、JupyterLab等。
  • Flexible。JupyterHub可以配置身份验证，以便提供对用户子集的访问。身份验证是可插入的，支持多种身份验证协议。
  • Scalable。JupyterHub容器友好，可以使用现代容器技术进行部署，可以跑在k8s上。
  • Portable。项目完全开源，且文档不少。

三、helm的配置

有了基本k8s环境，首先要安装helm包管理工具

官网提供的安装方法是直接curl：

curl https://raw.githubusercontent.com/helm/helm/HEAD/scripts/get-helm-3 | bash

然而由于目标路径在海外，不开代理的话基本curl不下来，而虚拟机上开全局代理比宿主机上开麻烦挺多，因此这里考虑绕开这个方法。

找了一大圈基本都要海外，最后找到一个别人写的get_helm的安装脚本，可是用这个脚本安装的helm版本是2.16，可是helm在v3做了较大改动。所以最后我采用的方法是用脚本安装2.16的helm，然后手动升级到最新版本。

后面发现可以直接添加helm对应的apt密钥，然后加上sourceslist，之后就可以通过apt装helm了，这个方法相对较简单：

curl https://baltocdn.com/helm/signing.asc | sudo apt-key add -
sudo apt-get install apt-transport-https --yes
echo "deb https://baltocdn.com/helm/stable/debian/ all main" | sudo tee /etc/apt/sources.list.d/helm-stable-debian.list
sudo apt-get update
sudo apt-get install helm

验证的话直接helm version，显示版本号基本就是成功了，**JupyterHub要求helm版本高于3.5**

四、JupyterHub搭建

官方文档：https://zero-to-jupyterhub.readthedocs.io/en/latest/jupyterhub/installation.html

4.1 Chart的实例化

首先在helm仓库中添加jupyterhub对应的chart：

helm repo add jupyterhub https://jupyterhub.github.io/helm-chart/
helm repo update

执行情况可通过helm search repo jupyterhub检查：

root@master1:~# helm search repo jupyterhub
NAME                     CHART VERSION    APP VERSION    DESCRIPTION                                       
jupyterhub/jupyterhub    1.1.3            1.4.2          Multi-user Jupyter installation

之后在工作文件夹初始化一个helm chart的配置文件config.yaml。对于jupyterjhub的config，不需要任何内容，因此新建一个空文件就行。

然后这里首先学习一些helm有关的前置知识：

  • 一个 Chart 是一个 Helm 包。它包含在 k8s 集群内部运行应用程序，工具或服务所需的所有资源定义。可以把它想像为一个自制软件，一个 Apt dpkg 或一个 Yum RPM 文件的 k8s 环境里面的等价物。

  • 一个 Repository 是 Charts 收集和共享的地方。有点像Fedora 软件包 repoFedora Package Database。

  • 一个 Release 是处于 Kubernetes 集群中运行的 Chart 的一个实例。一个 chart 通常可以多次安装到同一个群集中。每次安装时，都会创建一个新 release 。比如像一个 MySQL chart。如果希望在群集中运行两个数据库，则可以安装该 chart 两次。每个都有自己的 release，每个 release 都有自己的 release name。

这里官方建议的安装策略是：

helm upgrade --cleanup-on-fail \
  --install <helm-release-name> jupyterhub/jupyterhub \
  --namespace <k8s-namespace> \
  --create-namespace \
  --version=<chart-version> \
  --values config.yaml

这是一条比较集成的命令，大体是先helm install从前面提到的chart repository中install一个chart的实例，然后创建名字空间，指定chart版本。

可是直接这么干会failed，原因是拉不下来仓库的chart包，导致timeout

考虑到helm install可以直接install一个本地的chart目录，因此采取的方法是：宿主机上手动下载jupyterhub的helm-chart包，然后scp扔到集群所在的虚拟机上，在虚拟机本地install。这样能绕开从仓库里pull。具体做法：

宿主机上在https://jupyterhub.github.io/helm-chart/下载对应版本的`chart`包，本地解压，然后`scp`到相应目录，如：

scp -r ./jupyterhub root@master1:/root/jupyterhub

然后虚拟机上helm install：

helm install <releasename> ./jupyterhub

然后会发现还是timeout，不过我们现在已经得到了jupyterhub chart的目录结构了:

4.2 PV与PVC的创建

打了下日志看看，发现报错是pod has unbound PersistentVolumeClaims

原来是我之前对pvc这个概念没什么了解

一个应用肯定会跑多个pod，多个pod里面肯定会存放数据，当一个pod死了，数据也会跟着消失，因此k8s就有了pv的概念，pv就是持久化存储，相当于docker的volume，pvc就是分配pv的容量给pod。

在docker的设计实现中，容器中的数据是临时的，即当容器被销毁时，其中的数据将会小时，如果需要持久化数据，需要使用docker数据卷挂载宿主机上的文件或者目录到容器中，在kubernetes中，当pod重建的时候，数据也会丢失，kubernetes也是通过数据卷来挂载pod数据来实现持久化

在编写pvc配置文件时，如果不指定pv是谁，默认随机分配

确实，jupyterhub作为一个面向用户提供notebook平台，它肯定是要有持久化存储类似用户信息用户数据这些的.。

学习了一些pv的基本概念和对应yaml的写法之后，尝试创建一个pv：

kubectl create -f https://k8s.io/examples/pods/storage/pv-claim.yaml

这里直接用examples里面的yaml，稍作修改

再apply一个pvc：

kubectl create -f https://k8s.io/examples/pods/storage/pv-claim.yaml -n njh

发现这个pvc被成功绑定了：

NAME            STATUS    VOLUME           CAPACITY   ACCESSMODES   STORAGECLASS   AGE
task-pv-claim   Bound     task-pv-volume   10Gi       RWO           manual         30s

然后我就觉得这个pv能用了，修改了jupyterhub的values.yaml，把storageclass改成manual，尝试把hub-db-dir这个pvc绑定到刚才创建的pv

然后重新helm upgrade了一下，发现真的绑定成功了：

NAME            STATUS    VOLUME           CAPACITY   ACCESSMODES   STORAGECLASS   AGE
hub-db-dir      Bound     task-pv-volume   10Gi       RWO           manual         30s

可是发现pod现在变成一直CrashBackoff了，打了下log，发现error是打不开数据库文件：

jupyterhub (sqlite3.OperationalError) unable to open database file

然后文欣告诉我这里用的是NFS类型的StorageClass

4.3 NFS环境的搭建

​ NFS即网络文件系统，它允许网络中的计算机之间通过网络共享资源。将NFS主机分享的目录，挂载到本地客户端当中，本地NFS的客户端应用可以透明地读写位于远端NFS服务器上的文件，在客户端看起来，就像访问本地文件一样。

​ 用起来感觉就是每一个nfs client都有一个nfs server的挂载点。

4.3.1 NFS Server 的创建

可参考https://www.tecmint.com/install-nfs-server-on-ubuntu/

这里把nfs server部署在master2上，即33.33.33.81，部署过程如下：

首先装kernel-server：

apt install nfs-kernel-server
4.3.1.1 暴露nfs共享文件夹

指定nfs挂载目录，并删除目录权限中的所有限制，同时赋予所有人读写执行的权限：

mkdir -p /mnt/nfs_share
chown -R nobody:nogroup /mnt/nfs_share/
chmod 777 /mnt/nfs_share/

定义NFS服务器的访问权限，主要是修改/etc/exports的内容，修改为：

# /etc/exports: the access control list for filesystems which may be exported
#               to NFS clients.  See exports(5).
#
# Example for NFSv2 and NFSv3:
# /srv/homes       hostname1(rw,sync,no_subtree_check) hostname2(ro,sync,no_subtree_check)
#
# Example for NFSv4:
# /srv/nfs4        gss/krb5i(rw,sync,fsid=0,crossmnt,no_subtree_check)
# /srv/nfs4/homes  gss/krb5i(rw,sync,no_subtree_check)
#
/mnt/nfs_share  33.33.33.0/24(rw,sync,no_subtree_check)

解释一下几个字段：

  • /mnt/nfs_share ：nfs挂载目录

  • 33.33.33.0/24：暴露给哪些网段

  • rw：使能读写权限

  • sync：要求在应用更改之前将其写入磁盘

  • no_subtree_check：消除子树检查

暴露nfs共享文件夹，接着重启nfs内核服务以使能更改：

sudo exportfs -a
sudo systemctl restart nfs-kernel-server
4.3.1.2 利用ufw工具允许通过防火墙访问NFS

重新加载或启用防火墙，并检查防火墙的状态。

apt install ufw
ufw allow from 33.33.33.0/24 to any port nfs
ufw status
update-alternatives --list iptables
uname -r
ufw enable

4.3.2 创建NFS类型的StorageClass

有了NFS环境，接下来就能利用创建好的nfs server创建StorageClass了

这里基本按https://blog.csdn.net/boling_cavalry/article/details/105465672这篇博客来

首先创建rbac，改一下人家的yaml：

curl https://raw.githubusercontent.com/zq2599/blog_demos/master/storageclass-demo/rbac.yaml > ./rbac.yaml
sed -i ‘s/kafka-test/njh/’ rbac.yaml

主要是改成自己的名字空间，然后apply它：

kubectl apply -f rbac.yaml

创建provisioner，即关联NFS的工作类，负责给PVC提供存储资源，这里用的是nfs-client-provisioner

curl https://raw.githubusercontent.com/zq2599/blog_demos/master/storageclass-demo/deployment.yaml > ./deployment.yaml
sed -i 's/kafka-test/njh/' deployment.yaml

之后根据之前创建的nfs server编辑deployment.yaml文件，设置nfs参数：

    app: nfs-client-provisioner
  # replace with namespace where provisioner is deployed
  namespace: njh
spec:
  replicas: 1
  strategy:
    type: Recreate
  selector:
    matchLabels:
      app: nfs-client-provisioner
  template:
    metadata:
      labels:
        app: nfs-client-provisioner
    spec:
      serviceAccountName: nfs-client-provisioner
      containers:
        - name: nfs-client-provisioner
          image: quay.io/external_storage/nfs-client-provisioner:latest
          volumeMounts:
            - name: nfs-client-root
              mountPath: /persistentvolumes
          env:
            - name: PROVISIONER_NAME
              value: fuseim.pri/ifs
            - name: NFS_SERVER
              value: 33.33.33.81
            - name: NFS_PATH
              value: /mnt/nfs_share
      volumes:
        - name: nfs-client-root
          nfs:
            server: 33.33.33.81
            path: /mnt/nfs_share

然后创建它：

kubectl apply -f deployment.yaml

最后是创建StorageClass的脚本:

curl https://raw.githubusercontent.com/zq2599/blog_demos/master/storageclass-demo/class.yaml > ./class.yaml
kubectl apply -f class.yaml

可以看到挂载成功：

发现是selflink was empty这个问题。查看了k8s v1.20版本更新的官方文档，发现k8s在1.20版本弃用了selfLink，导致nfs后端无法自动创建pv，解决方法是手动加执行参数禁用掉这个remove，即添加参数--feature-gates=RemoveSelfLink=false

根据github上issue的解决方法尝试修改kube-apiserver.yaml，在其中添加上面提到的参数，却发现一直apply不了，而且本身也没有kube-apiserver.yaml这个配置文件。问了文欣，猜测可能跟我用的是kubeasz部署有关。于是教我直接在系统服务加参数

，具体说来即修改/etc/systemd/system/kube-apiserver.service，在kube-apiserver后面追加--feature-gates=RemoveSelfLink=false，然后重启服务。由于本集群有两个master，因此要在两个master上都要做此更改。
